Line-bound thread concepts for Vivi comments

These concepts keep the current Vivi workspace intact: left file tree, central open-file surface, right Review Queue inspector, and line-number based source comments. The model assumption is explicit: a thread is anchored to one line or line range. Rendered Markdown, rendered diff, and HTML preview may be where the reviewer notices the issue, but the thread still resolves to a canonical line range.

Must fix UI or implementation diverges from the existing line-thread model: invisible controls, stale composers, pending state ambiguity, or rendered comments that cannot be opened as a line/range thread.
Careful add New surface affordances that help users comment from rendered views. These should be introduced only when they preserve the current queue, line rail, and inspector vocabulary.
Do not do A separate comment product with its own shell, unrelated actions, or block-only model. That breaks the current mental model.
