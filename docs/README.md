# Documentation

Vivi documentation is grouped by the job a reader is doing.

- `product/`: product thesis, brief, requirements, non-goals, backlog, and UX
  acceptance criteria.
- `adr/`: accepted durable product or architecture decisions. Start here when
  asking why a direction was chosen.
- `contracts/`: public CLI, API, data, adapter, and comment lifecycle contracts.
- `architecture/`: package boundaries, runtime shape, security, performance,
  and backend design.
- `engineering/`: implementation plans, eval strategy, extension guides,
  Storybook lab notes, and local harness documentation.
- `runbooks/`: agent and maintainer operating procedures.
- `operations/`: install and operational distribution notes.
- `research/`: exploratory concepts and options that are not themselves
  implementation contracts.
- `ui-mocks/`: static visual and interaction concept artifacts.
- `release/`: release automation and formula material.
- `meta/`: documentation inventory and drift-control notes.

Use `contracts/` and `architecture/` for current behavior. Use `adr/` for
accepted decisions. Use `research/` for options, rejected ideas, and exploratory
language that should not be treated as the current contract.
